# Gonka new-host notifier

Бот без сервера: GitHub Actions по расписанию опрашивает список активных
участников сети Gonka и шлёт сообщение в Telegram-группу, если появился
новый хост. Состояние (кто уже был замечен) хранится прямо в репозитории
в файле `state/hosts.json`.

## Шаг 1. Создать Telegram-бота

1. Открой Telegram, найди `@BotFather`
2. Отправь `/newbot`, следуй инструкциям (имя, username)
3. BotFather пришлёт **токен** вида `123456789:AAExxxxxxxxxxxxxxxxxxxxxxxxxxxx`
   — сохрани его, он понадобится на шаге 4

## Шаг 2. Создать группу и получить chat_id

1. Создай в Telegram новую группу
2. Добавь в неё своего бота как участника
3. Отправь в группу любое сообщение
4. Открой в браузере:
   `https://api.telegram.org/bot<ТВОЙ_ТОКЕН>/getUpdates`
5. В ответе найди `"chat":{"id": -1001234567890, ...}` — это и есть `chat_id`
   (для групп он отрицательный, это нормально)

## Шаг 3. Создать репозиторий на GitHub

1. Создай новый **приватный** репозиторий (например, `gonka-host-bot`)
2. Залей в него все файлы из этой папки, сохранив структуру:
   ```
   gonka-host-bot/
   ├── .github/workflows/check-new-hosts.yml
   ├── new_host_bot.py
   ├── requirements.txt
   └── state/            (пустая папка — файл появится после первого запуска)
   ```

## Шаг 4. Добавить секреты в репозиторий

В репозитории: **Settings → Secrets and variables → Actions → New repository secret**

Добавь два секрета:
- `TELEGRAM_BOT_TOKEN` — токен из шага 1
- `TELEGRAM_CHAT_ID` — chat_id из шага 2

## Шаг 5. Первый запуск (вручную)

1. Перейди во вкладку **Actions** репозитория
2. Слева выбери workflow **"Check for new Gonka hosts"**
3. Нажми **Run workflow** (кнопка справа)
4. Дождись зелёной галочки — это значит скрипт отработал

⚠️ **Первый запуск ничего не пришлёт в Telegram** — это ожидаемо. Первый
раз бот просто сохраняет текущий список хостов как точку отсчёта (`state/hosts.json`
появится и закоммитится автоматически). Уведомления начнут приходить
начиная со второго запуска, когда будет с чем сравнивать.

## Шаг 6. Дальше — само расписание работает 24/7 без твоего участия

Workflow настроен на запуск каждые 30 минут (`cron` в файле workflow).
Ничего дополнительно поддерживать не нужно — GitHub Actions делает всё сам.

## Важная не до конца проверенная деталь

Скрипт написан с расчётом на то, что ответ эндпоинта
`http://node2.gonka.ai:8000/v1/epochs/current/participants` — это либо
голый список участников, либо словарь с этим списком под одним из
стандартных ключей (`active_participants` / `participants` / `result`).
Точная структура ответа живьём не проверялась.

**После первого ручного запуска (Шаг 5) обязательно:**
1. Открой лог запуска в Actions
2. Проверь, что нет ошибки вида `Could not find a participants list...`
3. Если ошибка есть — открой сам эндпоинт в браузере, посмотри реальную
   структуру JSON и поправь функцию `fetch_participants()` в
   `new_host_bot.py` под неё (нужная строчка явно об этом попросит).

То же самое касается функции `participant_id()` — сейчас она пробует
несколько вероятных названий поля (`participant_id`, `address`, `id`,
`inference_url`). Если сообщения в Telegram будут выглядеть как большой
JSON-дамп вместо короткого адреса/id — значит, ни одно из этих полей не
подошло, и в реальном ответе поле называется иначе. Поправить — одна
строчка в `participant_id()`.

## Как расширить (на будущее)

- Добавить в сообщение доп. детали о хосте (вес, модель) — как только
  будет понятна точная структура ответа
- Второй бот (алерты с дашборда) можно добавить как ещё один `.py` файл
  + ещё один job в том же workflow — не обязательно заводить отдельный репозиторий
